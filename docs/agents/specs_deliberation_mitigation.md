# Technical Specification — Deliberation Timeline & Mitigation Tools

This document defines the architecture, data schemas, and agent modifications for the **Deliberation Timeline** and **Simulated Tool Execution** features in **IncidentPilot**, utilizing the `agent-designer` skill.

---

## 1. Deliberation Timeline Architecture

To show real-time agent progression in the Streamlit UI, the `IncidentCoordinator` uses an **Event Stream (Generator)** model. To minimize latency, the calls to TriageAgent and RCAAgent run concurrently using a `ThreadPoolExecutor`.

### 1.1 State Definitions
The coordinator emits structured progress events:
```json
{
  "status": "started | parallel_start | processing | triage_done | rca_done | aggregating | complete",
  "message": "User-facing progress status message",
  "data": {}, // optional, contains Agent output payload
  "report": {} // present in complete event, containing the consolidated IncidentReport
}
```

### 1.2 Timeline Flow Matrix (Concurrent Execution)
1.  **Event Ingested** $\rightarrow$ Emit status: `"started"`, message: *"⚡ Coordinator starting incident processing..."*
2.  **Start Parallel Analysis** $\rightarrow$ Emit status: `"parallel_start"`, message: *"⚡ Initiating parallel analysis..."*
3.  **Trigger Concurrency** $\rightarrow$ Emit status: `"processing"`, message: *"🤖 Processing Triage and RCA concurrently..."*
4.  **Triage Complete (Async)** $\rightarrow$ Emit status: `"triage_done"`, message: *"✅ TriageAgent complete. Severity classified as P1."* (Sends triage payload).
5.  **RCA Complete (Async)** $\rightarrow$ Emit status: `"rca_done"`, message: *"✅ RCAAgent complete. Technical root cause identified and runbook tools mapped."* (Sends RCA payload).
6.  **Aggregation** $\rightarrow$ Emit status: `"aggregating"`, message: *"🔄 Consolidating reports..."*
7.  **Complete** $\rightarrow$ Emit status: `"complete"`, message: *"🎉 Incident response diagnosis completed successfully!"* (Sends the final `IncidentReport`).

## 2. Mitigation Skills & Tool Calling

We introduce simulated automation via high-risk recovery tools, exposed to the agents and executable by the user on the dashboard.

### 2.1 Tool Specifications

We expose two simulated DevOps tools inside `incidentpilot/app/core/tools.py`:

#### Tool 1: `simulate_restart_pod`
*   **Purpose**: Restarts a failing Kubernetes pod in a specific namespace.
*   **Parameters**:
    *   `pod_name` (string): Name of the pod to restart.
    *   `namespace` (string): Kubernetes namespace (defaults to `default`).
*   **Console Simulation Sequence**:
    ```
    [SHELL] kubectl delete pod payment-api-6fdf859664-xx28c -n default
    [INFO] Pod payment-api-6fdf859664-xx28c deleted successfully.
    [INFO] Kubelet creating new pod container...
    [INFO] Checking health check endpoint: http://payment-api/healthz...
    [SUCCESS] Status 200 OK returned. Pod is ready.
    ```

#### Tool 2: `simulate_kill_postgres_connections`
*   **Purpose**: Terminate deadlocked or idle-in-transaction PostgreSQL connections.
*   **Parameters**:
    *   `database_name` (string): Name of the target database.
    *   `min_idle_seconds` (integer): Filter connections idle for longer than this limit.
*   **Console Simulation Sequence**:
    ```
    [SHELL] psql -U postgres -d auth_db -c "SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE state = 'idle in transaction';"
    [INFO] Querying active connection count... Found 47 idle connections.
    [INFO] Terminating database connection backends (PID list: 4212, 4215, 4219)...
    [SUCCESS] Connection slots freed. Active connections decreased to 3/100.
    ```

## 3. Schemas & Agent Contract Changes

To bind these tools to the runbook steps, we update the data structures in `schemas.py`:

### 3.1 Runbook Step Binding
Each step in the runbook generated by the `RCAAgent` is now represented by a structured object rather than a simple string:
```python
from pydantic import BaseModel
from typing import Optional, List

class MitigationAction(BaseModel):
    tool_name: str  # "simulate_restart_pod" | "simulate_kill_postgres_connections" | "manual"
    parameters: dict # e.g. {"pod_name": "payment-api-xx", "namespace": "default"}

class RunbookStep(BaseModel):
    step_number: int
    instruction: str
    action: Optional[MitigationAction] = None

class RCAOutput(BaseModel):
    root_cause: str
    runbook: List[RunbookStep]
    confidence: float
```

## 4. Error Recovery & Safety Gates
*   **Pre-execution verification**: When the user clicks `🔧 Execute` in the UI, a confirmation modal is shown containing the generated parameters.
*   **Execution Logs**: Streamlit uses a code block block (`st.code`) styled as a dark CLI console to stream the tool output line-by-line using `time.sleep` for realistic timing.
